Add token-backed demo sessions, patient-record FHIR Bundle export, FHIR document Bundle export, FHIR Consent export, FHIR AuditEvent Bundle export, FHIR Provenance export for signed clinical documents, FHIR CapabilityStatement metadata, readiness checks, minimal consent store with revocation, Provider Directory, structured allergies, clinical conditions, service requests, workflow tasks, observations, diagnostic reports, imaging studies, medication workflows, inter-facility record transfer lifecycle milestones, and tamper-evident audit integrity checks.

This release note covers the product-grade hardening slice that replaced actor headers with Bearer sessions, added auth/RBAC API tests, exposed patient record collection bundles and document bundles with Composition, required an active patient consent matching the recipient before bundle export, added FHIR Consent export plus consent revocation metadata and API/UI controls that block later sharing, embedded Provider Directory and Consent resources so Organization/Practitioner/PractitionerRole/Endpoint/Consent references are resolvable, and added AllergyIntolerance/Condition/ServiceRequest/Task/Procedure/Observation/DiagnosticReport/ImagingStudy/MedicationRequest/MedicationDispense/MedicationAdministration/RecordTransfer domain/API/UI support for allergy safety alerts, diagnoses, service orders, execution tracking, procedures, problem lists, vital signs, laboratory results, diagnostic result reports, PACS/DICOM imaging metadata, medication orders, medication dispensing, medication administration, and record-transfer Task coordination. Signed clinical documents now expose FHIR Provenance so downstream systems can see which DocumentReference was finalized, by which practitioner, when the signing activity occurred, and which source attachment was used. RecordTransfer now has explicit send/receive milestones that audit status changes and map sent/received timestamps to FHIR Task.executionPeriod. Audit events are now sealed with SHA-256 payload and chain hashes, exposed through an audit-integrity API, a FHIR AuditEvent Bundle export for auditor review, and UI verification/export panels. The FHIR facade also publishes a CapabilityStatement metadata endpoint so integrators can discover supported R4 resources and prototype security assumptions. The API now has a `/ready` readiness endpoint and Docker uses it to wait for repository-backed dependencies before marking the service healthy.
